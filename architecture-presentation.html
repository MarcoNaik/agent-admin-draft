Overview

What is Struere?

A platform for building, deploying, and managing AI agents with built-in permission controls and environment isolation.

Permission-Aware
Every operation goes through RBAC with row-level scope rules and column-level field masks. No privileged data paths.
Environment Isolated
Full separation between development and production. Data, roles, configs, and permissions are all per-environment.
Real-Time
Native Convex subscriptions power live dashboards. No polling, no WebSocket management, no stale data.

Core Concept

Environment Isolation

Every piece of data, every role, every config exists independently in development or production. The environment is set at request entry (API key or dashboard toggle) and threaded through every operation.

development
production

The Environment type is defined in lib/permissions/types.ts as a union of "development" | "production".

Environment

Scoped vs. Shared

Per-Environment
  • entityTypes, entities, entityRelations
  • roles, policies, scopeRules, fieldMasks
  • agentConfigs
  • threads, events, executions, jobs
  • apiKeys, installedPacks
Shared Across Environments
  • agents (name, slug, description)
  • users, organizations
  • userOrganizations
  • toolPermissions
  • messages (scoped via thread)

Agents are shared but their configs are per-environment, looked up via the by_agent_env index on agentConfigs.

Security

Permission Engine

Every read, write, and tool call goes through a four-step permission pipeline. No shortcuts, no bypasses.

Row-Level Security
Scope rules filter which records an actor can see. A teacher only sees their own sessions.
Column-Level Security
Field masks hide sensitive fields. Allowlist strategy: new fields are hidden by default.
Deny Overrides Allow
If any policy says deny, access is blocked. Fail-safe model prevents accidental exposure.
Eager Resolution
ActorContext is built once per request with all role IDs resolved, filtered by the request's environment.

Security

The ActorContext

The central identity object threaded through every operation.

interface ActorContext { organizationId: Id<"organizations"> actorType: "user" | "agent" | "system" | "webhook" actorId: string roleIds: Id<"roles">[] // filtered by environment isOrgAdmin?: boolean environment: Environment // "development" | "production" }

Built by buildActorContext() in context.ts or buildActorContextForAgent() in agent.ts. System roles are resolved via by_org_isSystem index then filtered by environment.

16 / 22
Agents

Tool Identity Modes

When an agent calls a tool, whose permissions does it use?

inherit
Tool runs with the caller's ActorContext. Default mode. The agent or user's permissions apply.
system
Tool runs as a system actor with full access. Returns ActorContext with environment and isOrgAdmin: true.
configured
Tool runs with a specific role's permissions. Configured per-agent per-tool in toolPermissions.

Caveats

Known Limitations

Post-Index Filtering
entityRelations by_from/by_to and roles by_org_isSystem lack environment in the index. Filtered in memory.
Legacy Index Usage
sessions.ts, payments.ts, scheduling.ts still use by_org_slug/by_org_type instead of environment-aware indexes.
Relation Scope
Only field_match scope rules implemented. Complex relation-based scoping requires custom code patterns.
Event Payloads
Event payloads may contain unmasked historical data. No retroactive field mask application.
